1. What are Triggers?
A Trigger is a special type of stored procedure that automatically executes when an event occurs on a table or view.

Events include:

INSERT

UPDATE

DELETE

Example:
When an employee record is deleted, automatically save it to an audit table.

2. When do we use Triggers?
Triggers are used to:

Maintain audit logs

Validate data

Prevent invalid operations

Automatically update related tables

Enforce business rules

Example:
If someone deletes an employee, store the deleted employee details in EmployeeHistory.

3. Types of SQL Server Triggers
There are three types:

1. DML Triggers
Triggered by:

INSERT

UPDATE

DELETE

2. DDL Triggers
Triggered by:

CREATE

ALTER

DROP

3. Logon Triggers
Triggered when a user logs into SQL Server.

4. DML Triggers
A) AFTER Trigger
Runs after the operation is completed.

Example:

CREATE TRIGGER trgAfterInsert
ON Employee
AFTER INSERT
AS
BEGIN
    PRINT 'Employee Inserted'
END;
Flow:

INSERT → Data inserted → Trigger executes
B) INSTEAD OF Trigger
Runs instead of the original operation.

Example:

CREATE TRIGGER trgInsteadDelete
ON Employee
INSTEAD OF DELETE
AS
BEGIN
    PRINT 'Delete Not Allowed'
END;
Flow:

DELETE → Trigger runs → Row is NOT deleted
Difference
AFTER Trigger	INSTEAD OF Trigger
Executes after operation	Executes instead of operation
Data is modified first	Original operation is skipped
Used for logging	Used to prevent operations
5. Logon Trigger
Runs when a user logs into SQL Server.

Example uses:

Restrict login time

Log login details

Limit user access

6. Create Trigger
Syntax:

CREATE TRIGGER TriggerName
ON TableName
AFTER INSERT
AS
BEGIN
    -- SQL Statements
END;
7. Modify Triggers using SSMS
In SQL Server Management Studio (SSMS):

Expand Database

Expand Tables

Expand Triggers

Right-click Trigger

Select Modify

Edit SQL

Execute

8. DELETE Trigger
Runs whenever a row is deleted.

Example:

CREATE TRIGGER trgDelete
ON Employee
AFTER DELETE
AS
BEGIN
    PRINT 'Record Deleted'
END;
Can also copy deleted rows into an audit table.

9. Advantages of Triggers
Automatic execution

Enforces business rules

Maintains audit history

Improves data integrity

Prevents invalid changes

10. Disadvantages of Triggers
Can slow database operations

Difficult to debug

Hidden execution (users may not realize a trigger ran)

Can become complex with many triggers

11. What are Cursors?
A Cursor processes query results one row at a time.

Normally SQL works on all rows together, but a cursor lets you handle rows individually.

Example:

Employees

101
102
103
A cursor processes:

101
↓

102
↓

103
12. Life Cycle of a Cursor
The steps are:

Declare

Open

Fetch

Process

Close

Deallocate

Example:

DECLARE EmployeeCursor CURSOR FOR
SELECT Name FROM Employee;

OPEN EmployeeCursor;

FETCH NEXT FROM EmployeeCursor;

CLOSE EmployeeCursor;

DEALLOCATE EmployeeCursor;
13. Uses of SQL Server Cursor
Used when:

Processing rows one by one

Generating reports

Performing row-specific calculations

Executing complex business logic

14. Types of Cursors
Static Cursor
Creates a snapshot of the data.

Changes made after opening are not visible.

Dynamic Cursor
Reflects all changes (INSERT, UPDATE, DELETE) while open.

Forward-Only Cursor
Moves only from first row to last.

Faster than other cursor types.

Keyset Cursor
Shows updates to existing rows.

Does not show newly inserted rows.

15. Limitations of SQL Server Cursor
Slow performance

High memory usage

Consumes more resources

Not suitable for large datasets

More complex than set-based SQL

16. Alternatives to Cursor
Instead of cursors, prefer set-based operations, which are faster.

Common alternatives:

JOIN

GROUP BY

UPDATE

DELETE

MERGE

Window functions (ROW_NUMBER(), RANK())

WHILE loop (only when necessary)

Example:

Instead of updating one row at a time:

UPDATE Employee
SET Salary = Salary + 1000;
This updates all rows at once, making it much faster than a cursor.
