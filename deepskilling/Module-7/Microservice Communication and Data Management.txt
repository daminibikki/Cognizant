1. Inter-Service Communication in Microservices
In a microservices architecture, services often need to communicate with one another. The three most common communication methods are:

HTTP (REST APIs)

Messaging (Asynchronous)

gRPC

A. HTTP (REST API)
One service calls another using HTTP requests.

Example
Order Service
      │
HTTP Request
      │
Product Service
      │
HTTP Response
Example:

GET /api/products/101
The Product Service returns:

{
  "id": 101,
  "name": "Laptop",
  "price": 50000
}
Advantages
Simple to implement

Easy to understand

Works across different platforms

Widely used

Disadvantages
Slower than gRPC

Synchronous (caller waits for response)

B. Messaging (Asynchronous)
Instead of calling another service directly, a service sends a message to a message broker (such as RabbitMQ or Azure Service Bus).

Example
Order Service
      │
      ▼
 Message Broker
      │
      ▼
Notification Service
When an order is placed:

Order Service sends a message.

Notification Service receives it.

Customer gets an email or SMS.

Advantages
Loose coupling

Better reliability

Good for background processing

Doesn't require immediate response

Disadvantages
More complex

Harder to debug

C. gRPC
gRPC is a high-performance communication framework developed by Google.

It uses:

HTTP/2

Protocol Buffers (Protobuf)

instead of JSON.

Example
User Service
      │
    gRPC
      │
Payment Service
Advantages
Very fast

Smaller messages

Supports streaming

Excellent for service-to-service communication

Disadvantages
Harder to test manually

Less human-readable than JSON

HTTP vs Messaging vs gRPC
Feature	HTTP	Messaging	gRPC
Communication	Synchronous	Asynchronous	Synchronous
Data Format	JSON	Messages	Protobuf
Speed	Medium	Medium	Fast
Human Readable	Yes	Depends	No
Best For	Public APIs	Event-driven systems	Internal service communication
2. Service Discovery and Registration
In microservices, service locations can change as services are restarted or scaled.

Service Discovery allows services to find each other automatically.

Workflow
Product Service
      │
Registers
      ▼
Service Registry
      ▲
Searches
      │
Order Service
Examples of service registries:

Consul

Eureka

Kubernetes Service Discovery

Benefits
Automatic service lookup

Load balancing

Easier scaling

No hardcoded URLs

3. Implementing Communication in ASP.NET Core Web API
HTTP Communication using HttpClient
Example

var client = new HttpClient();

var response = await client.GetAsync(
    "https://localhost:5001/api/products");

var data = await response.Content.ReadAsStringAsync();
HttpClient is commonly used for REST communication.

Better Practice: IHttpClientFactory
builder.Services.AddHttpClient();
Controller

public class ProductController
{
    private readonly HttpClient _client;

    public ProductController(IHttpClientFactory factory)
    {
        _client = factory.CreateClient();
    }
}
Benefits:

Prevents socket exhaustion

Better performance

Centralized configuration

4. Database per Service Pattern
Each microservice owns its own database.

Example

User Service
      │
 User Database

Product Service
      │
 Product Database

Order Service
      │
 Order Database
Advantages
Independent services

Better security

Independent deployments

Technology flexibility

Disadvantages
Difficult joins across services

More complex reporting

Data consistency challenges

5. Shared Database Pattern
Multiple services use the same database.

User Service
      │
Product Service
      │
Order Service
      │
 Shared Database
Advantages
Easier reporting

Simpler joins

Easier transactions

Disadvantages
Tight coupling

Changes affect multiple services

Harder independent deployment

Database per Service vs Shared Database
Database per Service	Shared Database
Independent database	One common database
Loosely coupled	Tightly coupled
Better scalability	Limited scalability
Better for microservices	Better for monolithic applications
6. Entity Framework Core in Microservices
Each microservice has its own:

Models

DbContext

Database

Example

public class Product
{
    public int Id { get; set; }

    public string Name { get; set; }

    public decimal Price { get; set; }
}
DbContext

public class ProductDbContext : DbContext
{
    public ProductDbContext(
        DbContextOptions<ProductDbContext> options)
        : base(options)
    {
    }

    public DbSet<Product> Products { get; set; }
}
Register

builder.Services.AddDbContext<ProductDbContext>(options =>
    options.UseSqlServer(connectionString));
Each service manages only its own tables.

7. Data Consistency in Microservices
Since each service has its own database, maintaining consistency becomes challenging.

Example

Order Service
      │
Creates Order
      │
Payment Service
      │
Payment Failed
The order may already exist even though payment failed.

Solutions
Event-Driven Communication
Order Created
      │
Event
      ▼
Payment Service
      │
Payment Success
Services react to events instead of tightly coordinating with each other.

Saga Pattern (Concept)
A Saga manages a business transaction across multiple services.

Example:

Create Order

Reserve Product

Process Payment

If any step fails, compensate by undoing earlier steps (e.g., cancel the order or release inventory).

This avoids long distributed transactions.

8. Transactions in Microservices
Traditional database transactions across multiple services are generally avoided because they reduce scalability and increase coupling.

Instead, microservices often use:

Local transactions (within one service)

Events

Compensation actions (Saga pattern)

Eventual consistency

Eventual Consistency
Instead of making every database update happen instantly across all services, the system guarantees that all services will become consistent after some time.

Example

Order Created
      │
      ▼
Payment Processing
      │
      ▼
Inventory Updated
      │
      ▼
Notification Sent
Each step completes independently, and the overall system reaches a consistent state.
