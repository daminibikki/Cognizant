1.Single Responsibility
Rule:
A class should have only one job (one responsibility).
1. Testing – A class with one responsibility will have far fewer test cases.
2. Less dependency (lower coupling)
Each class is independent
So changing one class does not break others
3. Easy to find code (organization)
Small classes are easy to read and locate
No confusion like in big mixed classes
// example 
class Employee {
    String name;

    Employee(String name) {
        this.name = name;
    }

    public String getName() {
        return name;
    }
}

class EmployeePrinter {
    public void print(Employee emp) {
        System.out.println("Employee Name: " + emp.getName());
    }
}

public class Main {
    public static void main(String[] args) {
        Employee emp = new Employee("Damini");

        EmployeePrinter printer = new EmployeePrinter();
        printer.print(emp);
    }
}

Employee → Stores employee data.
EmployeePrinter → Prints employee details.
Each class has only one responsibility, so this follows the Single Responsibility Principle (SRP).

2. open/closed
Open-Closed Principle (OCP)
The Open-Closed Principle states that software entities (classes, methods, modules) 
should be open for extension but closed for modification.
This means that when new requirements arise,
we should add new functionality by extending existing code rather than changing already tested code. 
This reduces the risk of introducing bugs and makes the application easier to maintain.
Example: Instead of modifying the existing Guitar class to add a flame pattern, 
we create a new class SuperCoolGuitarWithFlames that extends Guitar. 
This adds new features without affecting the existing code.

class Guitar {
    private String make;
    private String model;
    private int volume;
}

class SuperCoolGuitarWithFlames extends Guitar {
    private String flameColor;
}

Benefit: Existing code remains stable, while new functionality can be added easily through inheritance or extension.


3. Liskov Substitution Principle (LSP)
The Liskov Substitution Principle states that a subclass should be able to replace its parent class without changing the correct behavior of the program.
In simple words, if one class inherits from another, it should behave in a way that users of the parent class expect. If the child class cannot properly support the parent's functionality, inheritance is not appropriate.
Example: A MotorCar can start an engine and accelerate, but an ElectricCar has no engine. If ElectricCar throws an error when turnOnEngine() is called, it breaks the expected behavior of the Car interface and violates LSP.
interface Car {
    void turnOnEngine();
    void accelerate();
}

class MotorCar implements Car {
    public void turnOnEngine() {
        System.out.println("Engine Started");
    }

    public void accelerate() {
        System.out.println("Car Accelerating");
    }
}

class ElectricCar implements Car {
    public void turnOnEngine() {
        throw new AssertionError("No Engine");
    }

    public void accelerate() {
        System.out.println("Electric Car Accelerating");
    }
}
Benefit: Following LSP makes code more reliable, reusable, and easier to maintain.


4.Interface Segregation
Interface Segregation Principle (ISP)
The Interface Segregation Principle states that a class should not be forced to implement methods it does not need. Instead of creating one large interface, it is better to divide it into smaller and more specific interfaces.
This helps keep code clean, flexible, and easier to maintain because classes only implement the functionality they actually use.
Example: A bear caretaker needs to wash and feed bears but does not need to pet them. By creating separate interfaces, each class can implement only the required methods.

interface BearCleaner {
    void washTheBear();
}

interface BearFeeder {
    void feedTheBear();
}

interface BearPetter {
    void petTheBear();
}

class BearCarer implements BearCleaner, BearFeeder {

    public void washTheBear() {
        System.out.println("Washing the bear");
    }

    public void feedTheBear() {
        System.out.println("Feeding the bear");
    }
}

class CrazyPerson implements BearPetter {

    public void petTheBear() {
        System.out.println("Petting the bear");
    }
}
Benefit: Classes only implement the methods they need, making the code simpler, more maintainable, and easier to extend.


5.Dependency Inversion Principle (DIP)
The Dependency Inversion Principle states that high-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces). This reduces tight coupling between classes and makes the code more flexible and easier to test.
Instead of creating objects directly inside a class, dependencies should be provided through interfaces. This allows us to replace implementations without changing existing code.

interface Keyboard { }

class StandardKeyboard implements Keyboard { }

class Monitor { }

class Windows98Machine {

    private Keyboard keyboard;
    private Monitor monitor;

    public Windows98Machine(Keyboard keyboard, Monitor monitor) {
        this.keyboard = keyboard;
        this.monitor = monitor;
    }
}

Benefit: Classes become loosely coupled, easier to test, maintain, and extend. New implementations can be added without modifying existing code.

conclusion:
The SOLID Principles are five object-oriented design principles that help developers write clean, maintainable, scalable, and flexible code.

S – Single Responsibility Principle
O – Open-Closed Principle
L – Liskov Substitution Principle
I – Interface Segregation Principle
D – Dependency Inversion Principle
By following these principles, software becomes easier to understand, test, maintain, and extend while reducing bugs and code complexity.
